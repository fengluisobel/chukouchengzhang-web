# 出口成章｜STT / TTS / LLM 选型调研（2026-03-30）

> 目的：为“出口成章”接入真实 STT / TTS / LLM 提供可执行的选型建议。
> 说明：本轮基于联网搜索摘要整理，适合做第一轮决策；最终接入前仍建议逐家核对官方价格、免费额度、协议兼容性与地区可用性。

---

## 结论先行

如果目标是：**先把 MVP 快速做成真可用，再逐步优化成本与效果**，建议分两层选型：

### 推荐组合（优先级最高）

#### 方案 A：最快落地
- **STT**：`faster-whisper / whisper.cpp` 本地自托管
- **TTS**：国内先用 **阿里云 Qwen3-TTS**，国外备选 **Google Cloud TTS** / **ElevenLabs**
- **LLM**：应用内做 **OpenAI-compatible 多 profile 快切**，优先接：
  - 海外：**OpenRouter**
  - 国内：**SiliconFlow / 兼容 OpenAI 格式的聚合或模型网关**

适合：
- 想先跑起来
- 不想被单一厂商锁死
- 想保留快速切换不同模型的能力

#### 方案 B：中文优先、国内体验优先
- **STT**：**FunASR**（本地）或 **讯飞**（云）
- **TTS**：**阿里云 Qwen3-TTS** 或 **讯飞**
- **LLM**：国内兼容 OpenAI 协议的提供商 / 自建网关（如 LiteLLM / New API）

适合：
- 中文场景很重
- 对国内访问稳定性更敏感
- 想降低海外网络和支付依赖

## 一、STT（Speech-to-Text）选型

### 1. 本地 / 开源方案

#### faster-whisper
优点：
- 基于 Whisper，精度高
- 推理速度和资源效率比原始 Whisper 更适合工程落地
- 适合服务端自托管
- 成本可控

缺点：
- 仍需自己部署和维护
- 音频预处理、并发与模型管理要自己做

适合：
- MVP 快速接入真实 STT
- 成本敏感
- 需要完全控制数据

#### whisper.cpp
优点：
- 纯 C/C++，轻量
- 适合本地、端侧、嵌入式和低依赖环境
- 非常适合以后考虑边缘部署

缺点：
- 工程接入比纯云 API 稍重
- 高并发服务端场景通常不如完整推理服务方便

适合：
- 本地部署
- 轻量运行环境

#### sherpa-onnx
优点：
- ONNX 生态，较轻量
- 实时性较好
- 适合多语言实时转录场景

缺点：
- 综合生态和现成案例不如 Whisper 系成熟

#### Vosk
优点：
- 超轻量
- 资源占用小

缺点：
- 精度通常不如 Whisper / FunASR

适合：
- 简单命令词识别
- 非核心场景

#### FunASR（中文非常值得重视）
优点：
- 中文场景友好
- 可本地部署
- 成本极低
- 很适合国内语音场景

缺点：
- 通用多语言生态不如 Whisper 系普适

适合：
- 中文优先产品
- 国内部署

---

### 2. 云 API / 商业方案

#### 讯飞
优点：
- 中文、方言、噪音场景表现强
- 免费额度相对友好
- 国内产品成熟

缺点：
- 更偏国内生态
- 若未来多语言重度扩展，灵活性不如 Whisper 自托管 + 多家云 API

#### 百度智能云
优点：
- 免费额度友好
- 成本较低
- 上手门槛相对低

缺点：
- 综合口碑和效果通常不是第一梯队首选

#### 阿里云 / 腾讯云
优点：
- 企业级稳定
- 云生态完整
- 适合标准 API 接入

缺点：
- 免费额度一般
- 若只做轻量 MVP，性价比未必最高

#### Google Cloud Speech-to-Text
优点：
- 多语言强
- 企业级稳定
- 海外生态成熟

缺点：
- 国内环境、支付与账号门槛更高
- 成本不算最低

#### Deepgram
优点：
- 实时性强
- 英文和实时语音交互场景很强
- API 体验成熟

缺点：
- 国内可用性、中文性价比需要实际再测

#### AssemblyAI
优点：
- 产品化程度高
- 附加能力多
- 免费额度有吸引力

缺点：
- 更偏完整语音理解平台，不一定是最便宜的 STT 起点

#### OpenAI Whisper API
优点：
- 多语言强
- 认知成本低
- 适合快速原型

缺点：
- 如果只是做 STT，本地 whisper/faster-whisper 很可能更便宜可控

---

### STT 建议

#### 如果现在就开做
1. **第一选择**：`faster-whisper` 本地服务
2. **中文备选**：`FunASR`
3. **云端快速起步**：`讯飞`（中文） / `Google STT`（多语言） / `Deepgram`（实时）

#### 对“出口成章”最实用的建议
- **MVP 第一刀**：本地 `faster-whisper`
- **中文加强刀**：做一个 `FunASR` provider 作为国内优先路线
- **云 API 兜底刀**：保留 `http provider`，后续可切 `讯飞` / `Deepgram` / `Google`

## 二、TTS（Text-to-Speech）选型

### 国外

#### ElevenLabs
优点：
- 情感和自然度非常强
- 免费层适合测试
- 适合演示效果拉满

缺点：
- 中文虽然在增强，但并不一定是中文性价比最优
- 付费后成本可能偏高

#### Google Cloud TTS
优点：
- 多语言强
- 免费层大
- 企业级稳定

缺点：
- GCP 使用门槛高于一些国内方案

#### Amazon Polly
优点：
- 稳定
- 全球化部署成熟

缺点：
- 体验上未必是“最好听”的第一选择

### 国内

#### 阿里云 Qwen3-TTS
优点：
- 低延迟
- 中文友好
- 音色克隆能力强
- 支持开源/本地部署路线

缺点：
- 新产品与新模型仍需实测稳定性

#### 讯飞 TTS
优点：
- 中文自然度好
- 稳定成熟

缺点：
- 某些新奇玩法不如新一代模型激进

#### 百度 TTS
优点：
- 永久免费额度友好
- 成本低

缺点：
- 综合体验未必是最优

#### Edge TTS（开源工具链）
优点：
- 免费
- 中文声音多
- 适合快速试音、快速演示

缺点：
- 更像工程捷径，不一定适合长期核心商用链路

---

### TTS 建议

如果目标是“免费又好用”：
- **国内优先**：`阿里云 Qwen3-TTS`、`讯飞`、`Edge TTS`
- **海外优先**：`Google Cloud TTS`、`ElevenLabs`

如果目标是“先做 Demo 效果”：
- `ElevenLabs`（展示效果）
- `阿里云 Qwen3-TTS`（中文实用）

如果目标是“长期可控”：
- 主线选 `阿里云 Qwen3-TTS` 或 `Google Cloud TTS`
- 演示效果备份 `ElevenLabs`

## 三、LLM 选型与“CC switch”式切换

### 目标
你提的关键需求是对的：

> 不能把 LLM 写死成一个供应商，必须能直接配置多个 url / key / model，然后快速切换。

这对 MVP 非常重要，因为：
- 不同模型在中文表达优化上的效果会差很多
- 不同供应商稳定性和速度波动很大
- 成本和速率限制会频繁变化
- 需要快速试错，而不是改代码才能切模型

---

### 当前最适合“出口成章”的做法

#### 做法 1：应用内直接支持多个 OpenAI-compatible profile
我已经按这个方向推进了实现底座。

特点：
- 每个 profile 配：`baseUrl` / `apiKey` / `model`
- 在 UI 或 API 中快速切换
- 不改业务代码即可换模型
- 当前最适合接入：OpenRouter、SiliconFlow、兼容 OpenAI 协议的网关或模型服务

适合：
- 快速迭代 MVP
- 做大量 A/B 测试
- 尽量减少耦合

#### 做法 2：上游再挂一个模型网关
例如：
- LiteLLM
- New API / One API
- Routerly
- LLM-API-Key-Proxy

好处：
- 统一把 OpenAI / Anthropic / Gemini / DeepSeek / 本地模型都变成一套入口
- 更适合团队化、预算控制、监控、fallback

适合：
- 后面模型越来越多
- 想把模型调度从业务应用里抽出去

---

### LLM 网关建议

#### 快速可用
- **OpenRouter**：适合先试多个海外模型
- **SiliconFlow / 国内兼容 OpenAI 的服务**：适合中文和国内访问

#### 自托管网关
- **LiteLLM**：功能最全，生态成熟，适合后期升级
- **New API / One API**：高性能、偏国产生态友好
- **Routerly**：适合需要更原生兼容 Anthropic 的情况

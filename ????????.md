- `/home/gem/workspace/agent/workspace/projects/出口成章/app/`

从“可演示 MVP”推进到“可实际使用的真能力 MVP”。

## 一、总体技术路线

建议拆成两层：

### 1）客户端层
可选：
- 现有 Web 原型继续作为业务验证前台
- SwiftUI iOS App 作为正式移动端

### 2）服务层
统一由后端提供接口：
- `POST /transcribe`
- `POST /polish`
- `POST /train/evaluate`
- `POST /ideas/archive`
- `GET /reports/daily`
- `GET /history`

建议原则：
- **客户端只负责采集、展示、交互**
- **表达理解、优化、评分全部后端统一处理**
- **避免把模型 prompt、密钥、策略散落在客户端**

## 二、分阶段接入顺序

## Phase 1：录音与转写接入
目标：用户真的能说话，不再靠文本模拟。

### 1.1 客户端录音能力

#### Web 原型
前端增加：
- `MediaRecorder`
- 音频格式优先：`audio/webm` 或 `audio/mp4`
- 录音时长限制：建议 1~180 秒
- 上传前展示：
  - 时长
  - 文件大小
  - 是否重录

#### iOS SwiftUI
使用：
- `AVAudioSession`
- `AVAudioRecorder`

建议参数：
- 采样率：16000 或 44100
- 单声道优先
- 输出格式：`m4a`

### 1.2 后端转写接口

接口：`POST /transcribe`

请求建议：
- `multipart/form-data`
- 字段：
  - `file`
  - `scene`
  - `userId`
  - `clientTs`

响应建议：
```json
{
  "text": "转写后的文本",
  "language": "zh-CN",
  "duration": 32,
  "wordCount": 128,
  "segments": [
    {
      "start": 0.0,
      "end": 4.2,
      "text": "我想做一个帮助用户提升表达的产品"
    }
  ]
}
```

### 1.3 STT 服务建议
优先级建议：

#### 方案 A：第三方云 STT（推荐）
适合先跑通 MVP。

要求：
- 中文普通话识别稳定
- 支持短音频实时或准实时转写
- 返回时长与分段结果
- 成本可控

#### 方案 B：自建 Whisper 类服务
适合后期。

优点：
- 数据可控
- 模型策略可调

缺点：
- 部署与 GPU 成本更高
- 维护复杂度更大

### 1.4 转写失败重试策略
- 首次失败：自动重试 1 次
- 第二次失败：记录错误码 + 提示用户重录或稍后再试
- 音频超过大小/时长限制：前端先拦截

## 三、表达优化能力接入
目标：把启发式本地逻辑替换成真实 LLM 优化服务。

### 3.1 接口定义
`POST /polish`

请求：
```json
{
  "rawText": "原始表达",
  "scene": "idea",
  "mode": "concise",
  "userId": "user_xxx"
}
```

响应：
```json
{
  "polishedText": "整理后的表达",
  "summaryTitle": "一句标题",
  "issues": [
    {
      "title": "句子偏长",
      "detail": "建议拆成 2~3 个短句"
    }
  ],
  "suggestedTags": ["表达训练", "产品表达"],
  "nextAction": "先用优化稿复述一遍"
}
```

### 3.2 模型侧要求
模型输出必须满足：
- 不改变用户原意
- 不把口语硬改成公文体
- 问题分析 3~5 条
- 给出下一步动作建议
- 输出结构稳定，建议使用 JSON schema 约束

### 3.3 Prompt 设计建议
输入维度：
- 场景 `scene`
- 风格 `mode`
- 原文 `rawText`

输出维度：
- 优化稿
- 标题
- 问题分析
- 标签
- 下一步建议

### 3.4 风险控制
- 增加兜底：LLM 返回格式错误时，自动触发一次“仅重排为 JSON”的补救请求
- 仍失败时：返回原文 + 轻量规则分析，不让主流程中断

## 四、训练评分能力接入
目标：从“能演示”升级到“有可信反馈”。

### 4.1 接口定义
`POST /train/evaluate`

请求：
```json
{
  "rawText": "原文",
  "polishedText": "标准稿",
  "attemptText": "用户复述稿",
  "round": 1,
  "scene": "idea",
  "userId": "user_xxx"
}
```

响应：
```json
{
  "clarityScore": 82,
  "structureScore": 79,
  "polishScore": 80,
  "feedback": "主线已经更清楚了，但开头结论还可以更硬一点。",
  "improvedPoints": [
    "复述已经覆盖核心价值",
    "结构比原文更稳定"
  ],
  "weakPoints": [
    "还缺少明确收束句"
  ]
}
```

### 4.2 建议评分逻辑
采用“两层评分”：

#### 第一层：规则分
检查：
- 是否覆盖核心观点
- 是否有结构词
- 是否过度发散
- 是否明显跑题

#### 第二层：模型分
评估：
- 清晰度
- 结构性
- 表达完整度
- 与标准稿的一致性

最终分数：
- `final = 规则分 * 0.35 + 模型分 * 0.65`

这样比纯模型口感更稳，也更容易调参。

### 4.3 训练反馈要求
训练反馈不要像考官批作文，要像教练：
- 先肯定进步
- 再指出最关键的 1~2 个问题
- 给下一轮可执行动作

## 八、验收顺序

### M2 验收标准（真能力 MVP）
- [ ] 能录真实音频
- [ ] 能上传音频并得到真实转写
- [ ] 能生成真实优化稿
- [ ] 能对复述稿生成真实评分
- [ ] 能归档到正式存储
- [ ] 能查看历史记录
- [ ] 主流程 1 次走通不崩

### 推荐验收脚本
1. 录入一段 20~40 秒真实表达
2. 系统返回真实转写
3. 系统生成优化稿与问题分析
4. 用户复述一遍
5. 系统返回评分与反馈
6. 点击归档
7. 在灵感库与报告页看到结果

## 结论
当前最对的推进方式不是先重做 UI，而是：

**保留现有主流程界面，先把录音、转写、优化、评分、归档这 5 条能力接成真的。**

这样一旦真能力跑通：
- Web 原型立刻变真 MVP
- SwiftUI 客户端也能直接复用同一套接口
- Demo、验收、上架准备会同时前进一步
